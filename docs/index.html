open-ltap · architecture notes

How reads and writes flow: the product (M0–M4) and the research track

The product: point one binary at the Postgres you already run and get fresh Delta tables on your own S3 — no Kafka, no Debezium, no new database platform. Storage-level integration with Neon is the research track: M5 (safekeeper source) is validated end-to-end against a local Neon stack, v2 is future work. Each diagram traces a write (W①…) to durable storage and the two kinds of reads (R①…): OLTP point queries and analytical scans.

The product · M0–M4 shipped

Any Postgres → Delta on your S3, one binary

The diagram below is the complete shipped pipeline — full DML (INSERT/UPDATE/DELETE via an append-only tombstone log), TOAST, COPY, initial snapshot, schema evolution, and inline compaction all live, on top of the M4 freshness read path — against RDS, self-hosted, Supabase, or Neon compute. It started as an INSERT-only slice at M0 (2026-07-02); M1–M4 (through 2026-07-06) deepened it into this without changing its shape. Writes commit exactly as before — Postgres acks after its local WAL fsync (W①), and the walsender streams the same WAL bytes to open-ltap in parallel (W②). OLTP reads (R①) never change: shared buffers, then local heap. Analytical reads (R②) scan Delta as of the last transcoded commit; the M4 tail merge (R③) upgrades that to read-your-writes.

You get

  • Any Postgres — RDS, Supabase, Neon, a laptop; no fork, no extension to install
  • Full DML, byte-exact — INSERT/UPDATE/DELETE (append-only tombstone log), COPY, TOAST, the whole type matrix
  • Schema tracks itself — ADD/DROP COLUMN, TRUNCATE/CLUSTER/rewrites, table create/drop/rename, all detected from the WAL
  • Read-your-writes — the M4 tail merge (R③) folds in-flight commits into every analytical read; ?min_lsn long-polling for hard RYW
  • Bounded forever — inline compaction + vacuum keep the change log, restart time, and storage from growing without limit
  • Transactional correctness & LSN lineage — commit/abort gate; every row stamped with its commit LSN

Costs — inherent to this shape

  • Two copies of table data — heap + Parquet; a single writer per Delta table
  • Costs Postgres one walsender — plus WAL retention if the transcoder stalls
  • Pre-image mirror lives in memory — one entry per live row; very large tables need RAM to match (the M5 oracle already answers misses there; full mirror-less operation is v2a territory)
  • Column type changes detach the table — Delta can't retype a column in place
  • lz4/zstd unsupported — wal_compression/default_toast_compression must stay off/pglz

Validated end-to-end · M5 — Neon source adapter

Same decoder, re-pointed at Neon's storage layer

For platform teams already running Neon — not a prerequisite for the product. Writes ack on a safekeeper Paxos quorum (W②); compute holds no durable state. OLTP reads hit the buffer pool, then fetch pages from the pageserver on a miss (R② GetPage@LSN). open-ltap consumes the safekeeper stream (W③) — downstream of compute, so transcoding costs Postgres nothing — and uses the pageserver as a random-access oracle: it can answer "what did page N of relation R look like at LSN X", which the sequential WAL tape can't — that's what UPDATE/DELETE pre-images, out-of-line TOAST chunks, and consistent backfill scans need (see Status below for what's actually wired up today). Note the honest trade-off: table data exists twice on S3 (layer files + Parquet) until v2.

Status: the safekeeper wire protocol (W③ — tenant/timeline packed into the libpq options startup param, JWT auth, slot-less START_REPLICATION) and decoding of Neon's custom heap rmgr are fully validated end-to-end against a live neon-compose stack (real Neon compute, PG 17.5): INSERT, UPDATE, DELETE, multi-insert/COPY, a forced post-checkpoint INSERT, and a genuine full-page-image restore all decoded byte-exact; an independent from-scratch client cross-checked WAL framing and CRC32C with zero mismatches against both the safekeeper and a vanilla walsender; and TOAST/DDL under this path uncovered two real bugs that turned out to be pre-existing and dialect-independent (reproduced identically on vanilla Postgres) — a TOAST decompression bug and a DDL-detection bug (pg_class/pg_attribute are mapped relations, so their stored relfilenode reads 0 on any Postgres) — both now fixed and re-verified. The pageserver oracle shown below is now wired into the engine too (2026-07-10): on the safekeeper path, UPDATE/DELETE pre-images are fetched live from the pageserver's pagestream_v3 protocol at the exact pre-record LSN — no pageinspect, no compute SQL involvement — verified live including a kill -9 + restart. What's still SQL: the initial snapshot/backfill (page-level visibility is a v2-scope research problem) and TOAST-chunk backfill for pre-toast-update rows (falls back to the Delta-rebuilt mirror). M5 is functionally complete for pre-images.

You get

  • Zero load on compute — the transcoder taps WAL downstream, at the safekeepers
  • Every table automatically — no replication slots, no publications, no per-table setup
  • The GetPage@LSN oracle — pre-images for UPDATE/DELETE, live and wired in; vanilla CDC has no equivalent random-access read
  • No Neon fork — an external subscriber; upstream upgrades stay painless
  • Read-your-writes analytics — pin LSN, scan Delta ≤ LSN, merge the tail (R④, with M4)

Missing / costs

  • Still two copies on S3 — Neon layer files and Parquet; the duplication only disappears in v2
  • You must run Neon — 3 safekeepers, pageserver, compute images, no managed control plane in OSS; a platform decision, not a tool install
  • Narrow audience — teams on RDS/self-hosted Postgres get nearly all the value from M0–M4 without this
  • Transcoder can lag — an independent consumer; freshness bounded by decode+commit unless the tail merge is deployed
  • Oracle covers pre-images only — initial snapshot/backfill still goes through the compute's SQL COPY (page-level visibility is a v2-scope research problem), and TOAST-chunk backfill for pre-toast-update rows is still unwired
  • Branches & PITR unmapped — Neon branches have no Delta-snapshot story yet

Future work · v2 — the research endgame

Transcoding inside compaction — columnar becomes canonical

What Databricks built, and the only design that eliminates the storage duplication. The write path is unchanged through the safekeepers, but the forked pageserver's compaction emits Parquet as the only durable copy (W④). The read paths invert: analytics scan canonical Parquet directly with an exact-LSN tail merge from the pageserver (R③, R⑤), while cold OLTP reads now pay the reverse path — a buffer-pool and page-cache miss means rebuilding the exact 8KB page from Parquet + WAL tail (R④), which is why the caching hierarchy has to be aggressive.

Status: the diagram below is the end state, not where things are today. The four fork-free probes (P0) are complete — an offline pageserver layer-file reader, deriving a table's schema straight from pg_class/pg_attribute pages with zero SQL, the GetPage@LSN oracle client (now also powering the M5 oracle above), and image-layer cadence measurement all validated live. The first stage, V2a — embedding the transcoder inside the pageserver — is underway: the engine has been extracted into a library, a fork of Neon is pinned to a specific commit, and a TranscodeSink tee scaffold plus several engine-adapter units are landing on a local fork branch. None of that has run the end-to-end gauntlet yet, and V2b/V2c — canonical Parquet, the reverse path described below — haven't started.

You get

  • One copy — columnar is the source of truth; storage cost halves, views can't diverge
  • Exact freshness for free — analytics pin an LSN and merge the unmaterialized tail from the pageserver (R⑤), zero Postgres involvement
  • Transcoding is amortized — it rides compaction work the storage layer already does
  • Everything from M0–M5 carries over — catalog tracking, visibility, TOAST, type fidelity are the same logic, relocated

Missing / costs

  • Pageserver fork — a rebase treadmill against fast-moving Neon internals, forever
  • The reverse path — rebuilding byte-addressed pages (indexes point at exact block/offset) is the research-grade problem; nobody outside Databricks has shipped it
  • No general OLTP hot tier — this diagram's page cache reconstructs pages on a miss, which is not the same as a keyed memtable, secondary indexes, in-place updates, or low-write-amp compaction; nothing here builds that, it's still Neon's own page cache doing the job
  • Cold OLTP reads pay for it — a cache miss costs Parquet→page reconstruction (R④); needs an aggressive cache hierarchy to hide
  • Shared failure domain — a transcoder bug now sits in the OLTP read path, not beside it
  • Branching, PITR, GC all rework — Neon's copy-on-write branches must map onto Delta snapshots
